Testing website security with ease!

What is this

xss_bomb is a API server that can be used to test website security, ex filtrate data from web pages, servers and in some cases not easily accessible content. With xss_bomb you can host a payload per user that can be ran on targeted web browsers to allow you to perform as many actions as you would want. You can share payloads across users on an instance. To allow for collaboration across penetration testers and bug bounty hunters. With each users dedicated customisable endpoint possibilities are endless between general request inspection code injection on the fly inside of dedicated endpoints and payload hosting.

With it's open source back-end xss_bomb allows you to easily understand and modify it's functionalities to allow a seamless penetration testing experience.

As of August 2023 I started working on xss_bomb 3 years ago originally as a mobile app to get notified when a payload would be triggered during a blind xss exploitation. Since the project has evolved a lot. I had many ideas and functionalities that I would have needed during either penetration tests or just overall testing things in general which is why now xss_bomb might look like a bit of a mess :) Hopefully this page should explain and clear up most of the confusion people have when I first showcase them this project. The source code is as usual available on github here. Like most of my projects this one included comes as is I do not intend this to be my full time job and is mainly a passion project If you wish to contribute it is obviously welcome and can be done through pull requests on github. I mainly advertise the back-end of xss_bomb and not the front-end (aka this website) since the website is not and never will be open source in the collaborative sense. I invite you to build your own front-end this one you are reading this page on is mainly here for demo purpose so not everything is perfect. But you can still use it by changing the server on the left to your own.

Navigation

Navigating through xss_bomb is made simple with the left panel that has links to all of the main pages.

You can also login/register and logout through the top right buttons.

Dashboard

The dashboard page is the main first page of xss_bomb and has all of the main information you will need to access with the endpoints that are linked to your account, the requests that are made to that endpoint and the websocket feed where you can retrieve the websocket ex filtrated data from it's respective endpoint. See endpoints for more information about the endpoints.

Endpoints

This page is used to generate custom endpoint links see more information here about how endpoints are used. With the endpoint generator you can fully customise the behaviour of your links if you need to quickly embed code inside of them on the fly here it can be done in a simple way.

My payload

Inside of the provided endpoints there is a specific code endpoint this endpoint is used to store a single file online and can be retrieved through the /code URI. This file can do anything you please and be used in any way possible for example you could combine a javascript script uploaded on the /code endpoint to ex filtrate data browser through your websocket link.

Shared payloads

On the my payload section there is an option to share a payload and this is then available publicly to every user of the instance inside of this page where you can click on a title to apply the code directly to your /code endpoint link.

My profile

This simple page exists to change your username and password for xss_bomb.

Endpoints

Each user has a token generated and assigned to him on each register. This token is present to allow requests made to the /api/token endpoint to be logged in to this specific user. These requests logged can then be fully inspected by the user after successful login.

The requests are shown on the left and on the right is the inspector. If a user clicks on the paper plane icon this will load the other requests present in the database (10 by 10 are loaded). A single user is provided a total of 5 different endpoints. Four of those are present on the main dashboard:

The endpoint with just the api token provided to the user after registration is the main endpoint with no specific features. This endpoint can be requested with (GET, POST, PATCH, PUT and DELETE) these requests can then be inspected in the inspector on the dashboard. The picture endpoint is a 1px by 1px picture that can be used on XSS with img html tags to view who accesses the invisible image. The /code endpoint is the one where your uploaded javascript file is stored. The last endpoint is the websocket endpoint where this can be used to upload bigger amounts of data to xss_bomb for ex filtration.

The last main endpoint is the /iframe endpoint and can be seen inside of the endpoint generator (found here). To correctly use this endpoint you need to provide a URL to where the iframe inside of the page is going to load from a title for the webpage and optionally code that can be injected in the page to control the behaviour of the iframe:
